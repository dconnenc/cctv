# Design Context

This file captures the design direction for CCTV (edinburgh-v1). Every frontend decision should reinforce these.

## Users

**Primary: the show operator.** Theater/improv producers, tech-booth runners, and event producers — the person with a laptop behind the sound board at a black-box venue, running an interactive show live. They are often alone, under time pressure, half-watching the stage, half-watching the UI. Mistakes are public; hesitation reads onstage.

**Secondary: the participants.** Today, audience members at the venue (phones, in-hand, venue wifi, varying light). Tomorrow, livestream viewers at home playing along.

**Job to be done:** Let the operator build an experience with calm confidence, then _drive_ it live with the fluency of a DJ or a lighting tech — so reacting to what's happening in the room feels like playing an instrument, not piloting software.

## Brand Personality

**Three words: comedy, experimental, modern.**

Not "funny" like clipart-and-exclamation-marks. Funny like a good show: confident, a little weird, willing to break the frame. Modern in craft (fast, precise, type-confident), not modern in vibe (not minimalist SaaS). Experimental means the interface is allowed to have opinions — an unexpected chrome, a specific typeface, a knowing reference — where most products play it safe.

**Voice reference points**

- **Jackbox** — saturated, party-game energy; readable from across the room; never sterile.
- **Dropout** — improv-gang warmth, in-joke confidence, bold headers, warm palette, clearly _made by the people in the videos_.
- **MySpace (productive reading only)** — user-expression over consistency, rough edges as a feature, the sense that a person made this.
- **Indie Chicago comedy** — iO, Second City, Annoyance, Hideout. Black box theaters, hand-photocopied zine flyers, sharpie on gaff tape, a CRT in the booth displaying a Q-list. This is the north star.

## Anti-References

Things this must _not_ look like:

- Generic SaaS dashboards (Linear-lite, Notion-lite, shadcn-default).
- Startup-purple gradient heroes and "Introducing ✨" marketing pages.
- Corporate event software (Eventbrite, Cvent, Whova).
- AI-output aesthetics: glowy cyan-on-dark, rounded-rectangle card grids, sparklines-as-decoration, gradient text.
- Zoom / Google Meet utility chrome.

If a stranger saw a screenshot, they should guess "a tool for making weird comedy shows," not "some YC B2B thing."

## Aesthetic Direction

The existing code already commits to a **retro-broadcast / tech-booth CRT** direction (phosphor green, Bebas Neue display, DM Mono body, Barlow Condensed UI, scanline and TV-glow effects in dark mode). Keep this. It's load-bearing for the indie-Chicago-comedy read — tech booths really do have old CRT monitors on them, Q-lists really are set in monospace, the phosphor glow is a genuine reference to broadcast/AV heritage, not decoration.

**Theme**

- **Default: system preference, falling back to dark.** Dark is the show-mode hero — operators work in low-light venues and it's where the CRT aesthetic is at full volume. Light is the daylight/planning mode — same personality, lower theatricality.
- Use `light-dark()` and data-theme attributes already in place; do not invent a third theme.

**Palette anchors (already defined in `app/frontend/styles.css` — use these, don't invent)**

- `--phosphor` (#c8f060) is the hero. Treat it like a rare accent: a live indicator, a primary action, a focus state — never as a default fill on everything. Its power is scarcity.
- `--amber` (#f5a623) for secondary warmth / warnings / "attention" states.
- `--red` (#ff4911) for destructive only.
- `--hot-white`, `--black`, `--deep`, `--dim` for surfaces. Tint neutrals toward phosphor when creating new surface steps.
- `--tv-glow` text-shadow on dark headings is a signature move — keep it, but audit contrast before shipping new surfaces.

**Typography (already selected — respect the stack)**

- Bebas Neue for display (show titles, big numbers, screen-top labels). All-caps is the house style here.
- DM Mono for body, timestamps, metadata, cue lists — the Q-list-on-a-CRT read.
- Barlow Condensed for UI chrome (buttons, tabs, small labels) — uppercase + wide letter-spacing.
- VCR_OSD_MONO is available; reserve for specific on-screen-display moments (countdowns, block IDs, status overlays). Do not use everywhere — it loses impact.

## Design Principles

These guide every design decision. When in doubt, return here.

1. **Operator-first, always.** The primary reader is one person in a dark booth during a live show. Optimize for: glanceability from 3 feet away, keyboard-reachable actions, no modals during live mode, state changes that never leave the operator guessing. If a decision would help a marketing screenshot but slow down the operator, choose the operator.

2. **Two modes, one personality.** _Setup/management_ is calmer, more spacious, readable in daylight on a couch. _Live/show_ is higher-contrast, denser, unambiguous, built for peripheral vision. Same typefaces, same palette, different rhythm and weight. The personality doesn't change between the two; the volume does.

3. **Tech booth, not dashboard.** Borrow from broadcast equipment, theater Q-lists, DJ decks, and patch bays — not from B2B SaaS. A status light beats a status badge. A typed cue number beats a labeled tag. An on-screen-display countdown beats a progress bar. Name things the way a stage manager would name them.

4. **Indie, not corporate.** Let the interface have fingerprints. Non-default typography, specific color, confident copy, the occasional knowing wink. Polish is mandatory; sterility is forbidden. If everything on screen could be swapped for a shadcn default without losing meaning, it's wrong.

5. **Phosphor is rare by design.** The hero green is an accent, not a fill. Use it to mean _something is live, something is yours to act on, something just changed._ A page where half the pixels are phosphor is a page that has forgotten how color hierarchy works.

6. **Accessibility is early, not retro-fit.** No hard WCAG target, but operators work in low-light venues with eyes adjusted to the stage — contrast failures hurt _us_, not just edge-case users. New surfaces ship with: ≥4.5:1 body contrast, keyboard navigation, visible focus states, reduced-motion fallbacks for the CRT/scanline effects, and text that remains legible with `text-shadow` disabled.

7. **Real-time is the medium.** All state comes from websockets. The UI's job is to feel _responsive to the room_ — an operator tapping a cue should see it land before they've lifted their finger; a participant's phone should update before they've put it back in their pocket. Never introduce manual refetches, loading spinners on actions, or optimistic states that can silently diverge from the server.
